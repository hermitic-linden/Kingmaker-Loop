# 谋主 Loop

***

## 设计哲学

猜想、讨论、试错、交接，AI的思考链长而乱，这很正常，人也如此。但是——

**不能把草稿纸献给主公。**

谋主 Loop 将构建一个轻量agent状态机循环，作为Codex、CC、Openclaw等Harness的可审计补丁，与其 Grill Me，或许不如白盒化AI，人类垂拱而治，随时介入。

***

## 文件结构

| 文件             | 角色      | 可否修改          |
| -------------- | ------- | ------------- |
| `谋主Loop.md`    | 系统规则    | ❌ 永不修改，放入提示词  |
| `ARCHITECT.md` | 运行时状态快照 | ✅ 可写的文件，放入工作区 |

***

## 执行模型

```
谋主Loop.md 解析规则
    → 读取 ARCHITECT.md 状态
    → 推导下一步操作
    → 产出变更
    → 人审 diff
    → 更新 ARCHITECT.md
```

***

## 快速开始

1. 把 谋主Loop.md 的内容放入 agent 提示词
2. 把 `ARCHITECT.md` 放到项目根目录
3. 与agent对话，让其首次定义项目
4. 逐层填充、推进、审查

***

## 设计 tradeoff 声明

### 1. 谋主不注册为 Skill —— 强制手动派遣

谋主 Loop **不该注册为 Skill**。这是有意为之的设计决策：

- 人在回路，而非AI 独用 ——有主公的谋士才有作为。

### 2. 无跨层互动逻辑 —— 鼓励各安职分

谋主 Loop **不内建多层ARCHITECT.md的互动逻辑**：

- **都管=都不管** —— 每个 ARCHITECT.md 只管自己这一层，不越权处理上级或下级的问题。
- **人类调度** —— 实际使用时，可以在项目的任意层级（不同大小文件夹）放置各自的 `ARCHITECT.md`，由人类决定何时、如何跨层协调。

这种设计让谋主在单一粒度上保持专注，同时保留了人类在多粒度间自由编排的能力。

### 3. ARCHITECT.md 不切分文件 —— 倒逼统筹全局

谋主 Loop **不将 ARCHITECT.md 拆分为多个文件**：

- **单一真相源** —— 所有状态、产出、转移记录集中在一个文件中，人类和 agent 都能一目了然地掌握全局
- **版本控制友好** —— 单次 diff 即可看清整轮 SOP 循环的全部变更

这种设计让全局状态始终可见，避免"只见树木不见森林"的执行盲区。
